# ESP-IDF Development Environment Setup

## What is ESP-IDF?

ESP-IDF (Espressif IoT Development Framework) is Espressif's official development framework for all ESP32 variants. It is a C/C++ framework built on top of FreeRTOS that provides:

- Direct hardware access with full control over all peripherals
- Production-grade WiFi/BT/BLE stacks
- Comprehensive driver library (SPI, I2C, UART, ADC, GPIO, timers, etc.)
- OTA update infrastructure
- Power management APIs
- Security features (secure boot, flash encryption, TLS)
- Filesystem support (SPIFFS, LittleFS, FAT)
- Partition table management

ESP-IDF is the most complete and up-to-date framework for ESP32. Arduino-ESP32 is actually built on top of ESP-IDF. If you need features not exposed by Arduino or PlatformIO, you'll need ESP-IDF directly.

### Version Guidance

- **v5.x** (current stable): Recommended for new projects. Major API changes from v4.x.
- **v4.4.x** (LTS): Still supported, widely documented. Some older examples target this.
- Always check which IDF version supports your chip. Newer chips (C6, H2) require v5.1+.

## Prerequisites by OS

### Linux (Ubuntu/Debian)

```bash
sudo apt-get update
sudo apt-get install git wget flex bison gperf python3 python3-pip python3-venv \
  cmake ninja-build ccache libffi-dev libssl-dev dfu-util libusb-1.0-0
```

For serial port access without sudo:
```bash
sudo usermod -a -G dialout $USER
# Log out and back in for this to take effect
```

### Linux (Fedora/RHEL)

```bash
sudo dnf install git wget flex bison gperf python3 python3-pip cmake ninja-build \
  ccache dfu-util libusb
```

### macOS

```bash
# Install Homebrew if not present
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

brew install cmake ninja dfu-util python3 ccache
```

No serial port driver needed for CP2102/CH340 on modern macOS (built-in).

### Windows

**Option A: Native (recommended)**
1. Download and run the ESP-IDF Tools Installer from https://dl.espressif.com/dl/esp-idf/
2. It installs Git, Python, CMake, Ninja, and the toolchains automatically
3. Provides an "ESP-IDF PowerShell" and "ESP-IDF Command Prompt" with environment pre-configured

**Option B: WSL2 (what you're likely using)**
- Follow the Linux instructions inside WSL2
- For USB serial access from WSL2, install usbipd-win on the Windows side:
  ```powershell
  # In Windows PowerShell (admin)
  winget install usbipd
  ```
- Then attach the serial device:
  ```powershell
  usbipd list              # Find your ESP32's bus ID
  usbipd bind --busid 1-3  # Replace with actual bus ID (one-time)
  usbipd attach --wsl --busid 1-3
  ```
- Inside WSL2 it will appear as `/dev/ttyACM0` or `/dev/ttyUSB0`

## Installation

### Step 1: Clone ESP-IDF

```bash
mkdir -p ~/esp
cd ~/esp
git clone --recursive https://github.com/espressif/esp-idf.git
cd esp-idf

# Check out a stable release (recommended)
git checkout v5.2.2  # or latest stable
git submodule update --init --recursive
```

To install a specific version without cloning the full history:
```bash
git clone -b v5.2.2 --recursive --depth 1 https://github.com/espressif/esp-idf.git
```

### Step 2: Install Tools

```bash
cd ~/esp/esp-idf
./install.sh esp32          # For original ESP32 only
# OR
./install.sh esp32,esp32s3  # For multiple targets
# OR
./install.sh all            # For all supported targets (takes longer, more disk space)
```

This downloads and installs:
- Xtensa and RISC-V cross-compilers
- OpenOCD for debugging
- Python dependencies in a virtual environment

### Step 3: Set Up Environment Variables

This must be done in every new terminal session:

```bash
. ~/esp/esp-idf/export.sh
```

Add an alias to your shell profile for convenience:
```bash
# Add to ~/.bashrc or ~/.zshrc
alias get_idf='. $HOME/esp/esp-idf/export.sh'
```

Then just type `get_idf` in any terminal to activate.

### Verify Installation

```bash
idf.py --version
```

## Project Structure

A typical ESP-IDF project looks like this:

```
my_project/
  CMakeLists.txt          # Top-level CMake file
  sdkconfig               # Configuration (generated by menuconfig)
  sdkconfig.defaults      # Default config overrides (checked into git)
  partitions.csv          # Custom partition table (optional)
  main/
    CMakeLists.txt        # Component CMake file for main
    main.c                # Entry point (app_main function)
    Kconfig.projbuild     # Custom menuconfig options (optional)
  components/             # Custom components (optional)
    my_driver/
      CMakeLists.txt
      my_driver.c
      my_driver.h
      Kconfig              # Component config options
  managed_components/     # Auto-downloaded components (do not edit)
```

### Top-Level CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.16)
include($ENV{IDF_PATH}/tools/cmake/project.cmake)
project(my_project)
```

### main/CMakeLists.txt

```cmake
idf_component_register(
    SRCS "main.c" "wifi_helper.c"
    INCLUDE_DIRS "."
    REQUIRES nvs_flash esp_wifi esp_event
)
```

The `REQUIRES` field lists ESP-IDF components your code depends on. Common ones:
- `driver` - GPIO, SPI, I2C, UART, ADC drivers
- `esp_wifi` - WiFi stack
- `esp_event` - Event loop
- `nvs_flash` - Non-volatile storage
- `esp_http_server` - HTTP server
- `esp_http_client` - HTTP client
- `mqtt` - MQTT client
- `spiffs` - SPIFFS filesystem
- `json` - cJSON library

### Minimal app_main

```c
#include <stdio.h>
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"
#include "esp_log.h"

static const char *TAG = "main";

void app_main(void)
{
    ESP_LOGI(TAG, "Hello from ESP-IDF!");

    while (1) {
        ESP_LOGI(TAG, "Running...");
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}
```

Note: `app_main` is called by the system after FreeRTOS is started. You can create additional tasks, but `app_main` itself runs as a FreeRTOS task. If it returns, the task is deleted (which is fine if you've spawned other tasks).

## Essential idf.py Commands

### Setting Target Chip

```bash
# Must be done BEFORE first build (sets toolchain)
idf.py set-target esp32      # Original ESP32
idf.py set-target esp32s2
idf.py set-target esp32s3
idf.py set-target esp32c3
idf.py set-target esp32c6
```

This regenerates `sdkconfig` for the target. If you switch targets, do a full clean first:
```bash
idf.py fullclean
idf.py set-target esp32s3
```

### Building

```bash
idf.py build
```

First build takes 2-10 minutes (compiles the entire framework). Subsequent builds are incremental and much faster (seconds to a minute).

Output is in `build/` directory:
- `build/my_project.bin` - Application binary
- `build/bootloader/bootloader.bin` - Bootloader
- `build/partition_table/partition-table.bin` - Partition table

### Flashing

```bash
idf.py -p /dev/ttyUSB0 flash
# or on macOS:
idf.py -p /dev/cu.usbserial-0001 flash
# or on Windows:
idf.py -p COM3 flash
```

Set default port to avoid typing it every time:
```bash
export ESPPORT=/dev/ttyUSB0
idf.py flash
```

### Serial Monitor

```bash
idf.py -p /dev/ttyUSB0 monitor
```

Keyboard shortcuts in monitor:
- `Ctrl+]` - Exit monitor
- `Ctrl+T Ctrl+H` - Show help
- `Ctrl+T Ctrl+R` - Reset target

### Build + Flash + Monitor (Combined)

```bash
idf.py -p /dev/ttyUSB0 flash monitor
```

This is the most common workflow during development.

### Configuration Menu

```bash
idf.py menuconfig
```

This opens a terminal-based configuration UI (ncurses). Key areas:
- **Serial flasher config** - Flash size, speed, mode
- **Partition Table** - Select partition scheme
- **Component config > WiFi** - WiFi settings
- **Component config > Bluetooth** - Enable/configure BT
- **Component config > FreeRTOS** - Tick rate, stack sizes
- **Component config > ESP System Settings** - CPU frequency, watchdog timers
- **Component config > Log output** - Default log level

Configuration is saved to `sdkconfig`. To create version-controllable defaults:

```bash
# Save only non-default values
idf.py save-defconfig
# Creates sdkconfig.defaults
```

### Other Useful Commands

```bash
idf.py size          # Show binary size breakdown
idf.py size-files    # Size per source file
idf.py size-components  # Size per component
idf.py erase-flash   # Erase entire flash (useful for NVS corruption)
idf.py erase-otadata  # Reset OTA boot selection
```

## Partition Tables

ESP32 flash is divided into partitions. The partition table defines the layout.

### Default Partition Table (no OTA)

| Name | Type | SubType | Offset | Size |
|------|------|---------|--------|------|
| nvs | data | nvs | 0x9000 | 0x6000 (24 KB) |
| phy_init | data | phy | 0xf000 | 0x1000 (4 KB) |
| factory | app | factory | 0x10000 | 1 MB |

This wastes most of a 4 MB flash. The factory partition only gets 1 MB.

### Custom Partition Table with OTA

Create `partitions.csv` in your project root:

```csv
# Name,   Type, SubType,  Offset,  Size,    Flags
nvs,      data, nvs,      0x9000,  0x5000,
otadata,  data, ota,      0xe000,  0x2000,
phy_init, data, phy,      0x10000, 0x1000,
ota_0,    app,  ota_0,    0x20000, 0x1C0000,
ota_1,    app,  ota_1,    0x1E0000,0x1C0000,
storage,  data, spiffs,   0x3A0000,0x60000,
```

Enable it in menuconfig:
- Partition Table > Partition Table > Custom partition table CSV
- Partition Table > Custom partition CSV file > `partitions.csv`

Or in `sdkconfig.defaults`:
```
CONFIG_PARTITION_TABLE_CUSTOM=y
CONFIG_PARTITION_TABLE_CUSTOM_FILENAME="partitions.csv"
```

### Partition Size Rules

- App partitions must be at least 64 KB and aligned to 0x10000 (64 KB boundary)
- Total of all partitions must fit in flash size minus the first 0x9000 bytes (reserved for bootloader + partition table)
- With 4 MB flash: ~4032 KB usable
- With OTA you need TWO app partitions of equal size

## Component Management

ESP-IDF has a component registry at https://components.espressif.com/ for reusable libraries.

### Adding Components

Create `main/idf_component.yml`:

```yaml
dependencies:
  espressif/led_strip: "^2.4.1"
  espressif/button: ">=2.0.0"
  espressif/esp_lvgl_port: "*"
```

Components are downloaded automatically on the next build into `managed_components/`.

### Using idf.py to Add Components

```bash
idf.py add-dependency "espressif/led_strip^2.4.1"
```

### Creating Your Own Components

```bash
idf.py create-component my_driver
# Creates components/my_driver/ with template files
```

A component's `CMakeLists.txt`:
```cmake
idf_component_register(
    SRCS "my_driver.c"
    INCLUDE_DIRS "include"
    REQUIRES driver esp_log
)
```

## sdkconfig.defaults Best Practices

Never commit `sdkconfig` to version control (it's huge and regenerated). Instead, use `sdkconfig.defaults` with only the settings you've changed:

```ini
# sdkconfig.defaults
CONFIG_ESPTOOLPY_FLASHSIZE_4MB=y
CONFIG_PARTITION_TABLE_CUSTOM=y
CONFIG_PARTITION_TABLE_CUSTOM_FILENAME="partitions.csv"
CONFIG_ESP_WIFI_SSID="MyNetwork"
CONFIG_ESP_WIFI_PASSWORD="MyPassword"
CONFIG_FREERTOS_HZ=1000
CONFIG_ESP_DEFAULT_CPU_FREQ_MHZ_240=y
CONFIG_ESP_SYSTEM_EVENT_TASK_STACK_SIZE=4096
CONFIG_LOG_DEFAULT_LEVEL_INFO=y
```

You can also have target-specific defaults:
```
sdkconfig.defaults           # Applied to all targets
sdkconfig.defaults.esp32     # Applied only for ESP32
sdkconfig.defaults.esp32s3   # Applied only for ESP32-S3
```

## Logging System

ESP-IDF provides a logging system with configurable levels:

```c
#include "esp_log.h"

static const char *TAG = "my_module";

ESP_LOGE(TAG, "Error: %s", error_msg);     // Red
ESP_LOGW(TAG, "Warning: value=%d", val);    // Yellow
ESP_LOGI(TAG, "Info: started");             // Green
ESP_LOGD(TAG, "Debug: buffer=%p", buf);     // (usually compiled out)
ESP_LOGV(TAG, "Verbose: byte=0x%02x", b);  // (usually compiled out)
```

Set per-tag log level at runtime:
```c
esp_log_level_set("my_module", ESP_LOG_DEBUG);
esp_log_level_set("wifi", ESP_LOG_WARN);  // Silence noisy WiFi logs
```

## Common Pitfalls

### 1. Forgetting to source export.sh
Every new terminal needs `. ~/esp/esp-idf/export.sh`. Without it, `idf.py` won't be found and toolchains won't be in PATH.

### 2. Wrong flash size
If your board has 16 MB flash but ESP-IDF defaults to 2 MB, your partition table will be wrong. Check and set in menuconfig: Serial flasher config > Flash size.

### 3. Stack overflow
FreeRTOS tasks have limited stack. Default is 3584 bytes. If you do heavy string processing, JSON parsing, or TLS, increase it:
```c
xTaskCreate(my_task, "my_task", 8192, NULL, 5, NULL);
```
Enable stack overflow detection in menuconfig: Component config > FreeRTOS > Check for stack overflow > Method 2.

### 4. Watchdog timer resets
Long-running loops without yielding trigger the task watchdog (default 5 seconds). Always call `vTaskDelay()` or `taskYIELD()` in loops.

### 5. GPIO conflicts with flash pins
On WROOM-32: GPIO6-11 are connected to the SPI flash. Using them crashes the chip. On boards with octal flash/PSRAM (some S3 modules), even more GPIOs are unavailable.

### 6. Not erasing NVS after changing partition table
If you change the partition layout, erase flash completely:
```bash
idf.py erase-flash
```

### 7. Running out of IRAM
If you see linker errors about IRAM, you have too much code in IRAM (interrupt handlers, functions marked IRAM_ATTR). Move non-critical code out of IRAM.

### 8. Mixing ESP-IDF versions
Don't build a project that was configured for v4.4 with v5.x tools without doing a `fullclean` and re-running `set-target`. API changes between major versions will cause confusing build errors.

### 9. Python virtual environment conflicts
ESP-IDF creates its own Python venv. If you have Anaconda or other Python managers active, they can conflict. Deactivate them before running `export.sh`.

### 10. Serial port permissions on Linux
If `idf.py flash` fails with "Permission denied" on `/dev/ttyUSB0`:
```bash
sudo usermod -a -G dialout $USER
# Then log out and back in
```

## VS Code Integration

Espressif provides an official VS Code extension: **Espressif IDF Extension**.

1. Install from VS Code marketplace
2. Run "ESP-IDF: Configure ESP-IDF Extension" from command palette
3. Point it to your ESP-IDF installation
4. It provides: build/flash/monitor buttons, menuconfig GUI, IntelliSense, debug support

Alternatively, use the C/C++ extension with a `c_cpp_properties.json` that points to ESP-IDF include paths (the build system generates `compile_commands.json` in the `build/` directory).
